# 추가금 결제 구조 초안

## 목표
- 현재 복지포인트 전용 구조를 유지하면서, 향후 `복지포인트 + 추가금` 결제를 붙일 수 있도록 데이터 구조를 먼저 정리한다.
- 실제 PG 연동과 결제 승인/취소는 마지막 단계에서 붙인다.

## 결제 모델
- `point_only`
  - 전액 복지포인트 결제
- `mixed_payment`
  - 일부는 복지포인트, 나머지는 외부 결제수단으로 결제
- 추후 확장
  - `card`
  - `kakao_pay`
  - `naver_pay`
  - `tosspay`

## 상품 구조
- 상품은 아래 2개 금액을 함께 가질 수 있어야 한다.
- `pointPrice`
  - 복지포인트 사용 금액
- `cashPrice`
  - 추가 결제 금액

예시:
- 상품 A: `49,000P + 0원`
- 상품 B: `20,000P + 15,000원`
- 상품 C: `0P + 39,000원`은 현재 범위에선 비권장

## 장바구니 구조
- 장바구니 아이템은 스냅샷으로 금액을 보관한다.
- 필요한 필드
  - `pointPriceSnapshot`
  - `cashPriceSnapshot`
- 장바구니 요약은 아래를 반환한다.
  - `itemPointAmount`
  - `itemCashAmount`
  - `shippingFeeCashAmount`
  - `discountCashAmount`
  - `finalPointAmount`
  - `finalCashAmount`

## 주문 구조
- 주문은 포인트 결제와 현금성 결제를 함께 표현할 수 있어야 한다.
- 필요한 필드
  - `usedPoint`
  - `additionalCashAmount`
  - `paymentMethod`
  - `paymentStatus`
  - `paymentSummary`

## 화면 구조
### 상품 상세
- 기본 표시
  - 상품명
  - 포인트 금액
- 추가금이 있는 경우만 표시
  - `추가 결제 예정 금액`

### 장바구니
- 주문 요약
  - 상품 금액(포인트)
  - 추가 결제 금액
  - 배송비
  - 총 결제 포인트
  - 총 결제 금액

### 주문서
- 결제수단 영역은 마지막 단계에서 활성화
- 현재는 구조만 준비
  - `결제수단 선택`
  - `카드/간편결제`
  - `결제 진행`

### 주문 상세
- 표시 항목
  - 사용 포인트
  - 추가 결제 금액
  - 결제수단
  - 결제상태

## 구현 순서
1. 타입과 API 응답 구조 확장
2. 상품/장바구니/주문 요약 계산 구조 반영
3. DB 컬럼 추가
4. 관리자 상품 등록 폼에 `추가금` 입력 추가
5. 프론트 주문서에 결제수단 UI 추가
6. PG 연동
7. 결제 승인/실패/취소/환불 처리

## 현재 반영 상태
- 프론트 타입에 `cashPrice`, `paymentSummary`, `paymentMethod`, `paymentStatus` 구조 추가
- 주문 checkout API payload에 `paymentMethod`, `cashAmount` 옵션 추가
- 백엔드 주문 응답에 추가금 결제 구조를 담을 수 있도록 기본값 처리 추가

## 마지막 단계에서 할 일
- Prisma schema에 실제 컬럼 추가
- seed 데이터에 추가금 상품 예시 추가
- 결제창 연동
- 주문 실패/부분취소/환불 정책 확정
