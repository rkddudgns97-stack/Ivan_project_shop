# 사내 복지몰 인증정책 변경안

## 한 줄 요약
사내 복지몰 회원가입은 사내 이메일 기반 인증코드 검증 방식으로 변경한다.

## 변경 배경
- 기존에는 로그인 상태가 고정된 프로토타입 형태여서 실제 사용자 유입 흐름이 없었다.
- 사내 복지몰 특성상 최소한 이메일 소유 확인은 필요하다.
- 향후 백엔드에서 메일 발송 서비스, 사내 계정 검증, 관리자 권한 분리를 붙이기 쉽도록 프론트 흐름을 먼저 정리한다.

## 정책 변경

### 기존
- 사용자는 이미 로그인된 상태로 진입한다.
- 회원가입 / 로그인 / 인증 단계가 없다.

### 변경
- 회원가입은 `이름 + 이메일 + 인증코드` 검증 후 완료한다.
- 로그인은 `이메일 + 인증코드` 검증 후 완료한다.
- 인증코드는 이메일로 발송되며 유효시간은 기본 5분으로 둔다.
- 이메일 인증이 완료된 계정만 복지몰 진입이 가능하다.
- 관리자 권한은 회원가입으로 부여하지 않고, 별도 운영 정책으로 관리한다.

## 사용자 플로우 변경

### 회원가입
1. 사용자가 회원가입 화면에서 이름과 이메일을 입력한다.
2. 인증 코드 발송을 요청한다.
3. 사용자는 이메일로 받은 인증 코드를 입력한다.
4. 코드 검증 성공 시 계정이 생성되고 자동 로그인된다.
5. 복지몰 홈으로 이동한다.

### 로그인
1. 사용자가 이메일을 입력한다.
2. 인증 코드 발송을 요청한다.
3. 인증 코드를 입력한다.
4. 검증 성공 시 로그인된다.

### 권한 분기
- 일반 가입 사용자는 `employee`
- 관리자 계정은 백오피스에서 별도 부여

## 화면 변경

### 신규 화면
- 인증 화면 `/auth`
  - 회원가입 탭
  - 로그인 탭
  - 인증 코드 요청
  - 인증 코드 입력

### 기존 화면 영향
- 메인 레이아웃 진입 전 인증 여부 확인 필요
- 관리자 페이지 진입 시 관리자 권한 체크 필요
- 헤더에 사용자명, 로그아웃, 관리자 전환 노출 규칙 반영 필요

## API 변경

### 신규 API
- `POST /auth/email-code`
  - 목적: 이메일 인증 코드 발송
- `POST /auth/signup`
  - 목적: 이메일 인증 기반 회원가입
- `POST /auth/login`
  - 목적: 이메일 인증 기반 로그인
- `GET /me`
  - 목적: 현재 로그인 사용자 조회

### 요청 예시
```json
{
  "email": "name@company.com",
  "purpose": "signup"
}
```

```json
{
  "name": "홍길동",
  "email": "name@company.com",
  "code": "123456"
}
```

## 데이터 모델 변경

### User
- `emailVerified: boolean`
- `authProvider: "email"`

### 고려 필요
- 실제 운영 시 `verificationRequests` 또는 `email_verification_tokens` 테이블 추가 필요
- 인증 시도 횟수 제한, 만료 처리, 재발송 제한 필요

## 백엔드 구현 메모
- 이메일 발송은 비동기 작업으로 분리하는 것이 안전하다.
- 인증 코드는 만료시간과 시도 횟수 제한이 필요하다.
- 관리자 권한은 회원가입 경로에서 절대 부여하지 않는다.
- 실제 운영에서는 사내 이메일 도메인 화이트리스트 정책을 검토해야 한다.

## 오픈 이슈
- 사내 이메일 도메인을 제한할지 여부
- 이메일 인증 후 추가 프로필 입력이 필요한지 여부
- 최초 가입 시 HR 마스터와 자동 매칭할지 여부
- 관리자 계정 생성 경로를 어디서 관리할지 여부
