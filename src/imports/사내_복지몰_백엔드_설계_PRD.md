# 사내 복지몰 백엔드 설계 PRD

## 한 줄 요약
사내 복지몰 백엔드는 임직원 복지 포인트, 상품, 주문, 배송, 관리자 운영을 하나의 정책과 데이터 모델로 일관되게 관리하는 운영 시스템이다.

## 목표 / 문제 정의

### 목표
- 임직원이 복지 포인트를 확인하고 상품을 구매할 수 있는 안정적인 백엔드 API를 제공한다.
- 포인트 지급, 차감, 환불, 조정을 모두 원장 기반으로 추적 가능하게 만든다.
- 관리자 운영에서 상품, 주문, 배송, 포인트 조정, 통계를 처리할 수 있게 한다.
- 향후 외부 제휴 상품 연동이 가능하도록 확장 가능한 도메인 구조를 만든다.

### 해결해야 할 문제
- 복지 포인트 잔액과 사용 이력이 불투명하면 신뢰가 떨어진다.
- 주문 취소, 반품, 환불 시 포인트 복구 기준이 애매하면 운영 이슈가 커진다.
- 사내 서비스 특성상 권한, 감사 로그, 정산 근거가 명확해야 한다.
- 외부 연동을 서둘러 넣으면 MVP 일정과 운영 복잡도가 급격히 올라간다.

## 가정 및 제약
- 인증은 회사 인트라넷 SSO 또는 사내 계정 시스템을 사용한다고 가정한다.
- 사용자 역할은 `employee`, `admin` 두 가지로 시작한다.
- 복지 포인트는 연 단위 지급이 기본이며, 지급/차감/환불/조정 이력은 모두 원장으로 남긴다.
- 상품 결제는 포인트 결제만 지원한다. 부족한 포인트는 별도 충전 기능으로 보완한다.
- 외부 제휴 상품 연동은 2차 단계로 두고, MVP는 자체 등록 상품 중심으로 운영한다.
- 배송은 자체 배송 또는 택배사 송장 등록 방식으로 처리하며, 실시간 택배 API 연동은 필수가 아니다.
- 관리자 화면 디자인은 별도 진행되므로, 본 문서는 화면별 백엔드 계약과 정책에 집중한다.

## 정책 설계

### 1. 자격 및 접근 정책
- 사내 계정으로 인증된 사용자만 복지몰을 사용할 수 있다.
- 재직 상태가 `active` 인 사용자만 상품 구매와 포인트 사용이 가능하다.
- 관리자만 상품 관리, 포인트 일괄 지급, 포인트 조정, 주문 상태 변경, 배송 정보 입력이 가능하다.
- 퇴사자 또는 휴직자 정책은 HR 원천 데이터 기준으로 상태 동기화 후 접근을 제한한다.

### 2. 포인트 정책
- 포인트는 `grant`, `recharge`, `use`, `refund`, `adjust_add`, `adjust_sub`, `expire` 타입의 원장 레코드로 관리한다.
- 잔액은 집계 값으로 보여주되, 진실의 원천은 원장이다.
- 연간 포인트 지급은 배치 작업으로 수행하며, 동일 지급 회차에 대한 중복 지급은 허용하지 않는다.
- 주문 생성 시 포인트를 즉시 차감하지 않고 `reserved` 상태로 예약한 뒤 결제 완료 시 `use` 로 확정한다.
- 주문 실패, 결제 만료, 관리자 취소 시 예약 포인트는 자동 해제한다.
- 포인트 충전은 회사가 허용한 결제 수단을 통한 충전 완료 후 `recharge` 로 적립한다.
- 유효기간이 있는 포인트는 만료일 기준으로 배치 만료 처리한다.

### 3. 상품 정책
- 상품은 `draft`, `active`, `inactive`, `sold_out` 상태를 가진다.
- `active` 상태이며 재고가 1 이상인 상품만 구매 가능하다.
- 상품 가격은 포인트 기준 정가로 관리한다.
- 상품 이미지는 외부 스토리지 URL 또는 사내 파일 스토리지 경로를 사용한다.
- 외부 제휴 상품은 `source_type = internal | partner` 로 구분한다.

### 4. 재고 정책
- 주문 확정 전 재고를 예약 차감하고, 결제 실패 시 원복한다.
- 단일 상품 다중 옵션을 고려해 `product_variant` 기준 재고를 관리한다.
- 재고 수량이 0이 되면 상품 또는 옵션은 자동으로 `sold_out` 처리할 수 있다.
- 관리자 재고 조정은 사유와 작업자를 반드시 남긴다.

### 5. 주문 / 배송 정책
- 주문 상태는 `created`, `paid`, `preparing`, `shipped`, `delivered`, `cancel_requested`, `cancelled`, `return_requested`, `returned` 를 사용한다.
- 장바구니 기준으로 1회 체크아웃 시 1개 주문과 N개 주문 아이템을 생성한다.
- 주문 생성 시 스냅샷 가격, 상품명, 옵션명, 배송지 정보를 저장한다.
- 배송 전 취소는 관리자 승인 없이 자동 취소 가능 여부를 정책으로 설정할 수 있다.
- 배송 후 취소는 반품 요청으로 전환한다.
- 배송 상태 변경과 송장 입력은 관리자만 가능하다.

### 6. 취소 / 반품 / 환불 정책
- `paid` 또는 `preparing` 상태에서 취소 요청 시 원칙적으로 취소 가능하다.
- 취소 확정 시 사용 포인트는 `refund` 원장으로 즉시 복구한다.
- `shipped` 이후에는 `return_requested` 로 접수하며, 반품 완료 후 환불한다.
- 부분 취소는 MVP 범위에서 제외하고 주문 단위 취소만 지원한다.
- 환불 완료 이후 중복 환불은 허용하지 않는다.

### 7. 관리자 운영 정책
- 포인트 조정, 상품 삭제, 주문 취소 확정, 재고 조정은 감사 로그를 남긴다.
- 관리자 화면의 주요 목록은 검색, 필터, 엑셀 다운로드를 지원하는 방향으로 설계한다.
- 통계는 실시간 집계보다 일배치 기반 요약 테이블을 우선 권장한다.

### 8. 외부 연동 정책
- HR 시스템과 사용자/재직 상태 동기화 배치를 둔다.
- 외부 제휴 상품 연동은 공급사 API 또는 정기 파일 업로드 배치로 분리 설계한다.
- 연동 실패 시 마지막 성공 시각과 실패 사유를 남기고 상품을 즉시 비활성화하지 않는다.

## 사용자 플로우

### 1. 복지 포인트로 상품 구매
1. 사용자가 복지몰 홈에서 상품을 탐색한다.
2. 상품 상세에서 옵션과 수량을 선택해 장바구니에 담는다.
3. 체크아웃 시 포인트 잔액과 주문 금액을 검증한다.
4. 주문 생성과 함께 포인트를 예약한다.
5. 주문 확정이 성공하면 포인트를 사용 확정하고 주문 상태를 `paid` 로 전환한다.
6. 관리자가 배송 준비, 배송 중, 배송 완료로 상태를 업데이트한다.

### 2. 포인트 충전
1. 사용자가 충전 화면에서 충전 금액을 선택한다.
2. 외부 결제 완료 콜백 또는 내부 결제 승인 이벤트를 수신한다.
3. 충전 내역을 저장하고 포인트 원장에 `recharge` 를 적재한다.
4. 사용자 잔액과 이력 화면에 즉시 반영한다.

### 3. 관리자 연간 포인트 일괄 지급
1. 관리자가 지급 회차와 대상 조건을 설정한다.
2. 서버는 대상자 스냅샷을 생성한다.
3. 중복 지급 여부를 검증한다.
4. 지급 작업을 비동기로 수행하고 결과 리포트를 남긴다.

### 4. 주문 취소 / 반품
1. 사용자가 주문 상세에서 취소 또는 반품을 요청한다.
2. 서버는 주문 상태와 취소 가능 조건을 검증한다.
3. 취소 확정 시 포인트를 환불하고 주문 상태를 `cancelled` 또는 `returned` 로 갱신한다.
4. 모든 변경 사항을 감사 로그와 주문 이력에 기록한다.

## 에지 케이스 / 예외 처리
- 포인트 부족 시 주문 생성 자체를 막고 최신 잔액 기준으로 재검증한다.
- 동시에 여러 주문이 발생하면 사용자 잔액과 재고를 트랜잭션으로 잠그거나 원자적으로 검증한다.
- 장바구니 담은 뒤 가격이나 재고가 바뀌면 체크아웃 시점 기준으로 다시 계산한다.
- 포인트 만료와 주문 결제가 동시에 일어나면 결제 요청 시점의 유효 포인트만 사용한다.
- 퇴사 처리된 사용자는 로그인 세션이 살아 있어도 주문/충전/취소 요청이 차단된다.
- 배송 완료 후 취소 요청은 자동 반려하고 반품 요청 흐름으로 유도한다.
- 관리자 수동 조정으로 잔액이 음수가 되지 않도록 검증한다.
- 외부 결제 콜백 중복 수신 시 멱등키로 중복 충전을 방지한다.
- 배치 지급 작업 재실행 시 지급 회차 + 사용자 조합의 유니크 키로 중복 지급을 막는다.
- 외부 상품 연동 실패 시 사용자에게는 최근 동기화 성공 기준 데이터를 노출하고 관리자에게만 경고를 보여준다.

## 화면 명세

### 1. 사용자 홈
- 목적: 잔액, 추천 상품, 카테고리 진입점 제공.
- 엔트리 포인트: 로그인 직후, 하단 탭 홈.
- 백엔드 필요 데이터: 사용자 프로필, 현재 잔액, 추천 상품 목록, 카테고리 목록, 공지.
- 주요 CTA: 상품 상세 진입, 카테고리 이동, 포인트 내역 이동.
- 상태: 정상, 추천 상품 없음, 공지 없음.

### 2. 상품 목록
- 목적: 검색, 필터, 정렬 기반 상품 탐색.
- 엔트리 포인트: 홈, 카테고리, 검색.
- 백엔드 필요 데이터: 상품 목록, 카테고리, 정렬 옵션, 필터 옵션, 페이지네이션.
- 주요 CTA: 상품 상세 이동.
- 상태: 결과 없음, 필터 결과 없음, 조회 실패.

### 3. 상품 상세
- 목적: 상품 정보 확인 및 장바구니 담기.
- 엔트리 포인트: 상품 목록, 추천 영역.
- 백엔드 필요 데이터: 상품 상세, 옵션, 재고, 이미지, 배송 정보, 관련 상품.
- 주요 CTA: 장바구니 담기, 바로 구매.
- 검증: 비활성 상품, 품절 옵션, 최대 구매 수량 초과.

### 4. 장바구니 / 체크아웃
- 목적: 주문 금액 확인과 최종 결제.
- 엔트리 포인트: 장바구니 아이콘, 바로 구매.
- 백엔드 필요 데이터: 장바구니 항목, 총 포인트, 배송지, 잔액, 정책 문구.
- 주요 CTA: 주문 생성, 항목 삭제, 수량 수정.
- 상태: 잔액 부족, 재고 부족, 배송지 없음.

### 5. 주문 목록
- 목적: 사용자의 주문 상태와 이력 확인.
- 엔트리 포인트: 마이페이지, 결제 완료 후.
- 백엔드 필요 데이터: 주문 목록, 상태 필터, 기간 필터.
- 주요 CTA: 주문 상세 이동.
- 상태: 주문 없음, 조회 실패.

### 6. 주문 상세 / 배송 추적
- 목적: 주문 아이템, 상태, 송장, 취소/반품 요청 제공.
- 엔트리 포인트: 주문 목록.
- 백엔드 필요 데이터: 주문 상세, 상태 이력, 배송 정보, 취소 가능 여부, 반품 가능 여부.
- 주요 CTA: 취소 요청, 반품 요청.
- 검증: 이미 취소됨, 반품 기간 초과.

### 7. 포인트 지갑 / 이력
- 목적: 잔액, 지급, 사용, 환불, 충전 이력 확인.
- 엔트리 포인트: 홈 잔액 카드, 마이페이지.
- 백엔드 필요 데이터: 현재 잔액, 원장 목록, 지급 회차 정보, 만료 예정 포인트.
- 주요 CTA: 충전 화면 이동.
- 상태: 이력 없음.

### 8. 포인트 충전
- 목적: 부족한 포인트를 충전.
- 엔트리 포인트: 포인트 지갑.
- 백엔드 필요 데이터: 충전 정책, 허용 금액 단위, 결제 수단 목록.
- 주요 CTA: 충전 요청.
- 검증: 최소/최대 충전 금액, 결제 실패, 중복 콜백.

### 9. 관리자 대시보드
- 목적: 운영 현황 요약.
- 엔트리 포인트: 관리자 로그인 후 첫 화면.
- 백엔드 필요 데이터: 총 사용자 수, 사용률, 주문 현황, 미처리 요청, 재고 경고.
- 주요 CTA: 상품 관리, 주문 관리, 포인트 관리 이동.

### 10. 관리자 상품 관리
- 목적: 상품 등록, 수정, 비활성화, 재고 관리.
- 엔트리 포인트: 관리자 메뉴.
- 백엔드 필요 데이터: 상품 목록, 상태, 카테고리, 재고, 제휴 구분.
- 주요 CTA: 등록, 저장, 비활성화, 재고 조정.
- 검증: 필수값 누락, 가격 음수, 이미지 누락.

### 11. 관리자 임직원 / 포인트 관리
- 목적: 사용자 조회, 포인트 지급, 조정, 일괄 업로드.
- 엔트리 포인트: 관리자 메뉴.
- 백엔드 필요 데이터: 사용자 목록, 재직 상태, 현재 잔액, 지급 이력, 조정 이력.
- 주요 CTA: 일괄 지급, 개별 조정, 엑셀 업로드.
- 검증: 중복 지급, 조정 후 음수 잔액.

### 12. 관리자 주문 / 배송 관리
- 목적: 주문 처리와 배송 상태 갱신.
- 엔트리 포인트: 관리자 메뉴.
- 백엔드 필요 데이터: 주문 목록, 상세, 상태, 송장 정보, 취소/반품 요청.
- 주요 CTA: 상태 변경, 송장 등록, 취소 승인, 반품 승인.
- 검증: 허용되지 않은 상태 전이, 이미 환불 완료된 주문.

### 13. 관리자 제휴 / 연동 관리
- 목적: 외부 상품 연동 상태와 실패 이력 관리.
- 엔트리 포인트: 관리자 메뉴.
- 백엔드 필요 데이터: 파트너 목록, 마지막 동기화 시각, 성공/실패 건수, 장애 로그.
- 주요 CTA: 수동 동기화, 파트너 비활성화.
- 상태: 연동 없음, 동기화 실패.

## PRD

### 기능 개요
사내 복지몰 백엔드는 복지 포인트 기반 상품 구매와 관리자 운영을 지원하는 REST API 및 배치 시스템으로 구성된다.

### Objective
- 임직원 셀프서비스 구매 경험을 안정적으로 지원한다.
- 포인트와 주문의 정합성을 보장한다.
- 운영팀이 최소 인력으로 관리 가능하게 한다.

### Scope
- 사용자 인증 연동
- 사용자/재직 상태 관리
- 포인트 원장 및 잔액 조회
- 연간 일괄 지급
- 포인트 충전
- 상품 / 카테고리 / 옵션 / 재고 관리
- 장바구니 / 주문 / 주문 상세
- 주문 취소 / 반품 / 환불
- 배송 상태 관리
- 관리자 운영 API
- 기본 통계 및 감사 로그

### Out of Scope
- 외부 쇼핑몰 실시간 주문 전송
- 복수 법인 / 복수 조직별 포인트 정책
- 부분 취소 / 부분 반품
- 쿠폰, 프로모션, 타임딜
- 실시간 택배 추적 API
- 정산 회계 시스템 완전 자동 연동

### User Story
- 직원으로서 나는 내 복지 포인트 잔액과 사용 이력을 쉽게 확인하고 상품을 사고 싶다.
- 직원으로서 나는 배송 전에는 주문을 취소하고 싶다.
- 직원으로서 나는 포인트가 부족하면 추가 충전하고 싶다.
- 관리자로서 나는 연간 포인트를 일괄 지급하고 예외 케이스를 조정하고 싶다.
- 관리자로서 나는 상품과 재고를 운영하고 주문 배송 상태를 업데이트하고 싶다.

### 기능 요구사항

#### 사용자 API
- 내 프로필 조회 API
- 포인트 잔액 / 이력 조회 API
- 포인트 충전 요청 / 결제 콜백 API
- 카테고리 조회 API
- 상품 목록 / 상세 조회 API
- 장바구니 CRUD API
- 주문 생성 API
- 주문 목록 / 상세 조회 API
- 주문 취소 요청 API
- 반품 요청 API

#### 관리자 API
- 관리자 대시보드 요약 API
- 상품 등록 / 수정 / 비활성화 API
- 재고 조정 API
- 사용자 조회 API
- 포인트 일괄 지급 API
- 포인트 수동 조정 API
- 주문 목록 / 상세 / 상태 변경 API
- 송장 등록 API
- 취소 승인 / 반려 API
- 연동 상태 조회 API

#### 배치 / 워커
- HR 사용자 동기화 배치
- 연간 포인트 일괄 지급 배치
- 포인트 만료 배치
- 외부 연동 상품 동기화 배치
- 통계 요약 배치

### 비기능 요구사항
- 모든 금전성 변경은 트랜잭션으로 처리한다.
- 관리자 민감 작업은 감사 로그를 남긴다.
- 주요 API는 멱등성 키를 지원한다.
- 장애 시 주문과 포인트 정합성이 깨지지 않아야 한다.
- 사용자 응답 시간 목표는 일반 조회 기준 1초 이내를 목표로 한다.
- 개인정보는 최소 수집하고 배송 정보 등 민감 데이터는 암호화 저장을 고려한다.

### 정책 요구사항
- 포인트 원장은 삭제하지 않고 정정 원장으로만 보정한다.
- 주문 상태 전이는 서버에서만 통제한다.
- 퇴사자 상태 동기화 이후 신규 주문을 차단한다.
- 만료 포인트는 사용 가능 포인트 계산에서 제외한다.

### 화면 요구사항
- 사용자 화면은 잔액, 이력, 주문 상태가 항상 최신 데이터로 보여야 한다.
- 관리자 화면은 검색/필터/정렬 기반 운영이 가능해야 한다.
- 취소/반품 가능 여부는 버튼 노출 전에 서버 응답으로 판단한다.

### 이벤트 / 상태 로직

#### 포인트 원장 상태 흐름
- `grant` -> 사용 가능 잔액 반영
- `reserved` -> 주문 생성 시 임시 차감
- `use` -> 주문 결제 확정
- `refund` -> 취소/반품 환불
- `expire` -> 만료 배치 차감

#### 주문 상태 전이
- `created -> paid`
- `paid -> preparing`
- `preparing -> shipped`
- `shipped -> delivered`
- `paid|preparing -> cancel_requested -> cancelled`
- `shipped|delivered -> return_requested -> returned`

### 의존성
- 사내 인증 시스템
- HR 사용자 마스터
- 파일 스토리지
- 알림 채널 이메일 또는 사내 메신저
- 결제 연동이 필요한 경우 PG 또는 사내 결제 수단

### 리스크
- 포인트 회계 정책이 늦게 확정되면 데이터 모델 변경 가능성이 크다.
- 외부 연동을 MVP에 포함하면 일정과 운영 난도가 크게 상승한다.
- 관리자 수기 조정이 많아지면 감사 로그와 승인 정책이 추가로 필요해질 수 있다.
- 취소/반품 정책을 단순화하지 않으면 주문 상태 설계가 복잡해진다.

### 권장 개발 순서
1. 인증 연동, 사용자/권한 모델, 포인트 원장 구축
2. 상품 / 카테고리 / 재고 API 구축
3. 장바구니 / 주문 / 포인트 차감 트랜잭션 구축
4. 관리자 주문 / 배송 / 포인트 운영 API 구축
5. 배치, 통계, 외부 연동 확장

### 권장 기술안
- 애플리케이션: NestJS 또는 Spring Boot
- 데이터베이스: PostgreSQL
- 캐시 / 락: Redis
- 파일 저장: S3 호환 스토리지
- 비동기 작업: BullMQ 또는 메시지 큐 기반 워커
- 문서화: OpenAPI / Swagger

## 오픈 질문 / 확인 필요 사항
- 포인트 유효기간은 지급일 기준 1년인지, 회계연도 말 일괄 만료인지 확인이 필요하다.
- 포인트 충전이 실제 개인 결제인지, 급여 공제인지 확인이 필요하다.
- 취소는 배송 전 자동 승인인지, 관리자 승인형인지 확정이 필요하다.
- 반품 가능 기간과 반품 승인 주체를 정해야 한다.
- 외부 제휴 상품 연동을 MVP에 넣을지 2차로 뺄지 결정이 필요하다.
- 관리자 권한을 `admin` 하나로 둘지, `ops`, `finance` 등 세분화할지 확인이 필요하다.
- 배송지 정보 저장 범위와 개인정보 보존 기간 정책이 필요하다.
